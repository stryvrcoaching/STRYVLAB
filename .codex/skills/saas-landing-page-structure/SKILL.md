# SaaS Landing Page Structure Skill

## Purpose

Use this skill to build or evaluate a SaaS landing page with a clear functional structure.

The landing page must guide the visitor through this sequence:

> Capture attention → clarify the problem → prove credibility → explain the solution → reduce objections → convert.

The goal is not to create a visually attractive page only. The goal is to create a page where every section has a conversion function.

---

## When to use this skill

Use this skill when the task involves:

- creating a SaaS landing page;
- refactoring an existing SaaS landing page;
- improving SaaS homepage conversion structure;
- turning a product brief into a landing page;
- auditing whether a landing page has the right sections;
- generating landing page copy and layout sections;
- building a reusable landing page template for a SaaS product.

Do not use this skill for:

- e-commerce product pages;
- personal portfolios;
- long-form sales letters;
- marketplace category pages;
- blog articles;
- documentation pages;
- pure brand pages without conversion intent.

---

## Required product inputs

Before implementing the landing page, identify or infer the following:

1. Product name
2. Target user
3. Main painful problem
4. Desired result
5. Core product mechanism
6. Primary CTA
7. Secondary CTA, if needed
8. Pricing model or access model
9. Available proof:
   - customer logos;
   - testimonials;
   - usage numbers;
   - reviews;
   - case studies;
   - beta users;
   - founder credibility;
   - product demo.
10. Product maturity:
   - early-stage;
   - self-serve SaaS;
   - B2B demo-led SaaS;
   - enterprise SaaS;
   - complex platform.

If proof is missing, do not invent it. Use honest placeholders in the code only if the user explicitly asked for placeholders. Otherwise, write the section so it can support future real proof.

---

## Core rule

A SaaS landing page sells value before features.

A feature describes what the product does.

A benefit describes what the user gets.

Bad:

> Advanced dashboard with custom filters.

Better:

> Spot the accounts that need attention before they churn.

Every feature section must be translated into a user-facing outcome.

# Canonical SaaS landing page structure

## 1. Sticky navbar

### Function

Make navigation simple and keep the main CTA accessible.

### Required elements

- Logo
- Anchor links to major sections
- Primary CTA

### Common nav links

- Product
- Features
- How it works
- Pricing
- FAQ
- Resources, optional

### Rules

- Keep it simple.
- Make it sticky on scroll.
- Keep the CTA visible on desktop.
- Use a clean mobile menu.
- Do not overload the nav with too many links.
- Do not introduce multiple unrelated CTAs.

---

## 2. Hero section

### Function

Communicate the value proposition immediately.

The visitor must understand in a few seconds:

- what the product does;
- who it is for;
- what problem it solves;
- what result it helps achieve;
- what action to take.

### Required elements

1. Eyebrow
2. Headline
3. Supporting paragraph
4. Primary CTA
5. Secondary CTA, optional
6. 2 or 3 short benefits
7. Initial proof or reassurance
8. Product visual

### Headline formula

Use a result-oriented headline.

Possible formulas:

> [Achieve desired result] without [main pain].

> The [product category] for [target user] who need to [job to be done].

> Turn [painful current state] into [desired future state].

### Supporting paragraph formula

> [Product name] helps [target user] achieve [desired result] by [core mechanism].

### Product visual

Prioritize:

1. Real product screenshot
2. Realistic product mockup
3. Short product animation
4. Annotated workflow visual
5. Abstract illustration only as a last resort

### Mistakes to avoid

- Vague headline
- Generic “future of work” claim
- No visible product
- CTA below the fold
- Too much text
- Feature list without outcome

---

## 3. Initial social proof

### Function

Reduce doubt early.

### Possible formats

- Customer logos
- User count
- Team count
- Review rating
- Short testimonial
- Security/compliance badge
- Media mention
- Beta/waitlist credibility
- Founder or domain credibility

### Rule

Proof must be real.

Never invent:

- customers;
- logos;
- usage numbers;
- reviews;
- testimonials;
- ratings;
- media mentions;
- compliance claims.

If the SaaS has no proof yet, use honest credibility:

- built with pilot users;
- currently in private beta;
- designed from real user workflows;
- product walkthrough;
- founder expertise;
- transparent roadmap.

---

## 4. Problem section

### Function

Make the visitor recognize their current pain.

The problem must be specific, not generic.

### Structure

1. Problem title
2. Short context paragraph
3. List of concrete pains

### Strong problem examples

> Your team is not slow. Your workflow is fragmented.

> The issue is not the lack of tools. It is the lack of a system.

> Leads are not lost randomly. They are lost in the handoff.

### Pain examples

- Data is scattered across tools.
- Context is hard to retrieve.
- Manual follow-ups are missed.
- Reporting takes too long.
- Teams make decisions with incomplete information.
- Customers do not see clear progress.
- Repetitive tasks consume strategic time.

### Mistakes to avoid

- Fake problem
- Overdramatic copy
- Generic pain
- Too many abstract claims
- Talking about the product too early

---

## 5. Pain vs relief section

### Function

Show the before/after clearly.

Use a two-column comparison.

### Structure

Column 1: Without the product  
Column 2: With the product

Each pain must map to a corresponding relief.

Example:

Without:
- Manual reporting every week.

With:
- Reports are generated automatically from live data.

Bad mapping:

Pain:
- Too much manual work.

Relief:
- Modern interface.

Better mapping:

Pain:
- Too much manual work.

Relief:
- Repetitive actions are automated.

---

## 6. Short testimonial or proof block

### Function

Insert proof after the visitor recognizes the problem.

### Format

Use a short, specific testimonial.

Include, when available:

- quote;
- name;
- role;
- company;
- photo;
- measurable result.

### Rules

- Keep it short.
- Tie it to the previous problem.
- Prefer specific outcomes over generic praise.
- Do not invent testimonials.

Bad:

> Great tool, highly recommend.

Better:

> We replaced three spreadsheets and reduced weekly reporting time by 40%.

---

## 7. Solution section

### Function

Present the SaaS as the logical answer to the problem.

### Structure

1. Solution title
2. Simple explanation
3. 3 to 5 key benefits
4. Product visual or workflow diagram

### Explain what the product does in outcome language

Cover:

- what it centralizes;
- what it automates;
- what it clarifies;
- what it accelerates;
- what it makes more reliable.

### Rule

Keep this section simple.

Do not explain the full product architecture here.

---

## 8. How it works section

### Function

Make the product feel easy to understand and adopt.

### Recommended format

Three steps.

Example:

1. Connect  
   Import data, connect tools, or create the workspace.

2. Configure  
   Set workflows, rules, users, permissions, or goals.

3. Execute  
   Track, automate, analyze, deliver, or collaborate from one place.

### Rules

- Use user-facing steps.
- Avoid technical implementation steps.
- Show the shortest credible path to value.
- Add one objection reducer if true.

Examples of objection reducers:

- No credit card required.
- Works with your existing tools.
- No technical setup required.
- Setup takes less than 10 minutes.
- Cancel anytime.

Only use these if they are true.

---

## 9. Intermediate CTA

### Function

Allow already-convinced visitors to act before the end of the page.

### Structure

- Short value statement
- Primary CTA
- Microcopy reassurance

### Rule

Keep the CTA consistent with the hero CTA.

Do not switch between unrelated actions such as “Start free trial”, “Book demo”, and “Download PDF” unless the business model requires it.

---

## 10. Product demo section

### Function

Reduce uncertainty by showing the product in action.

### Acceptable formats

- Demo video
- GIF
- Interactive product tour
- Annotated screenshots
- Product walkthrough
- Before/after workflow visual

### Video guideline

If using a video, keep it under 3 minutes.

### Demo content

Show:

1. Starting point
2. Main action
3. Visible result
4. User benefit

Example for support SaaS:

1. Ticket arrives
2. Ticket is prioritized
3. Ticket is assigned
4. Team resolves and reports faster

### Rule

The demo must show the product or workflow. It must not be only a branding video.

---

## 11. Storytelling / credibility section

### Function

Explain why this product or team is credible.

### Possible angles

Founder-led:

- founder experienced the problem;
- founder observed the same pain repeatedly;
- founder built the product to solve it.

Company-led:

- team has domain expertise;
- team worked with the target users;
- product is based on repeated customer workflows;
- company has credible experience in the space.

### Structure

1. Short title
2. One to three concise paragraphs
3. Optional founder/company visual
4. Optional link to About page

### Rules

- Keep it short.
- Connect the story to the customer problem.
- Avoid ego-driven storytelling.
- Avoid long autobiographical content.

---

## 12. Pricing or access section

### Function

Clarify cost, commitment, or access path.

Choose the format based on SaaS type.

## Self-serve SaaS

Show pricing plans directly.

Each plan should include:

- plan name;
- target user;
- price;
- main benefit;
- key features;
- limitations;
- CTA.

CTA examples:

- Start free trial
- Get started
- Try for free

## B2B demo-led SaaS

Use a demo or sales-led section.

Possible CTAs:

- Book a demo
- Talk to sales
- Request pricing

Clarify what happens after the CTA.

## Early-stage SaaS

Use:

- waitlist;
- private beta;
- early access;
- founder plan;
- limited access.

Do not fake traction.

## Pricing rules

- Do not create fake urgency.
- Do not use fake discounts.
- Do not invent “was/now” pricing.
- Lifetime deals should not be default.
- The best plan can be highlighted if there is a real reason.
- The pricing section must reduce friction, not create confusion.

---

## 13. FAQ / objections section

### Function

Answer the objections that may block conversion.

FAQ is not secondary content. It is rational sales support.

### Objection categories

Compatibility:

- Does it work with my tools?
- Can I import my data?
- Is it suitable for my industry?

Setup:

- How long does setup take?
- Do I need technical skills?
- Can I migrate from another tool?

Pricing:

- Is there a free trial?
- Can I cancel?
- What is included in each plan?

Security:

- Where is data stored?
- Who can access the data?
- Is the product compliant with required standards?

Differentiation:

- Why not use a spreadsheet?
- Why not use Notion, Airtable, Zapier, Slack, or a competitor?
- What makes this product different?

Maturity:

- Is the product in beta?
- What features are available now?
- What is coming next?

### Rules

- Answer directly.
- Keep answers short.
- Do not dodge difficult objections.
- Do not claim compliance or certifications unless verified.

---

## 14. Social proof wall

### Function

Strengthen confidence before the final CTA.

### Possible content

- testimonials;
- customer logos;
- review snippets;
- case studies;
- user numbers;
- screenshots of real feedback;
- media quotes;
- security badges.

### Rule

Quality beats quantity.

Three credible testimonials are better than a wall of vague praise.

Never use fake testimonials or fake avatars.

---

## 15. Final CTA

### Function

Convert the visitor after the page has clarified, proven, explained, and reassured.

### Structure

1. Result-oriented headline
2. Short paragraph
3. Primary CTA
4. Optional secondary CTA
5. Reassurance microcopy

### Rules

- Do not introduce a new idea.
- Repeat the main value proposition.
- Use the same CTA logic as the rest of the page.
- Make the next step obvious.

---

## 16. Footer

### Function

Provide useful, legal, and institutional links.

### Recommended content

- logo;
- short baseline;
- product links;
- resources;
- pricing;
- contact;
- social links;
- privacy policy;
- terms of service;
- legal notice;
- security or compliance link if relevant.

### Rule

The footer must be complete, clean, and unobtrusive.

# SaaS type adaptations

## Self-serve SaaS

Prioritize:

- free trial;
- clear pricing;
- simple onboarding;
- product demo;
- low-friction CTA.

Best CTA examples:

- Start free trial
- Try for free
- Create account

## Demo-led B2B SaaS

Prioritize:

- credibility;
- ROI;
- use cases;
- security;
- integrations;
- proof;
- demo booking.

Best CTA examples:

- Book a demo
- Talk to sales
- Request pricing

## Early-stage SaaS

Prioritize:

- clear positioning;
- honest credibility;
- product walkthrough;
- beta access;
- waitlist;
- user feedback.

Best CTA examples:

- Request access
- Join waitlist
- Get early access

Never invent traction.

## Complex platform SaaS

Prioritize:

- simplified homepage;
- segmentation by use case;
- strong navigation;
- feature pages;
- pricing page;
- security page;
- documentation.

Do not overload the main landing page with every detail.

# Copywriting checklist

The landing page must answer:

1. Who is this for?
2. What problem does it solve?
3. What result does it create?
4. How does it work?
5. Why is it credible?
6. What does it cost or how can the user access it?
7. What action should the user take now?

Headlines must be:

- specific;
- short;
- result-oriented;
- understandable without context;
- tied to the section function.

Paragraphs must be:

- concise;
- direct;
- concrete;
- free from unnecessary jargon.

CTA copy must be:

- visible;
- consistent;
- action-oriented;
- value-oriented when possible.

Avoid:

- “Revolutionize your workflow”
- “The ultimate platform”
- “AI-powered magic”
- “Boost everything”
- “10x your business”
- unsupported revenue claims
- vague productivity claims
- fake urgency

---

# Design checklist

The landing page must have:

- clear visual hierarchy;
- product visible early;
- strong above-the-fold section;
- readable typography;
- enough spacing;
- mobile-first responsiveness;
- accessible contrast;
- visible CTA states;
- no purely decorative section with no conversion role.

Above the fold must contain:

- clear headline;
- supporting paragraph;
- CTA;
- product visual;
- proof or reassurance.

Mobile version must:

- show the value proposition quickly;
- keep CTA accessible;
- avoid unreadable product screenshots;
- stack sections cleanly;
- preserve performance.

---

# Technical implementation rules

When implementing this landing page:

- Respect the existing project stack.
- Do not add dependencies unless necessary.
- Build sections as clean reusable components.
- Avoid one giant page component when possible.
- Separate content constants if useful.
- Use semantic HTML.
- Use accessible buttons and links.
- Add alt text for images.
- Optimize images.
- Lazy-load below-the-fold media.
- Keep animations lightweight.
- Preserve Core Web Vitals.
- Avoid placeholders in production.
- Do not invent customer proof.

---

# Minimum required sections

A SaaS landing page generated with this skill must include at least:

1. Navbar
2. Hero
3. Social proof or honest credibility
4. Problem
5. Solution
6. How it works
7. Product demo or product visual
8. Pricing, access, or demo booking
9. FAQ
10. Final CTA
11. Footer

Optional but recommended:

- Pain vs relief
- Intermediate CTA
- Storytelling / credibility
- Social proof wall

---

# Acceptance criteria

The landing page is acceptable only if:

1. The value proposition is clear within a few seconds.
2. The target user is identifiable.
3. The main problem is specific.
4. The solution is explained simply.
5. Benefits are expressed as outcomes, not only features.
6. The product is shown concretely.
7. The proof is real or honestly framed.
8. Main objections are handled.
9. CTA logic is consistent.
10. Pricing or access path is clear.
11. The page is responsive.
12. The code is maintainable.
13. No fake proof, fake urgency, or unsupported claim is used.
14. Every section has a clear conversion role.
15. The page follows the sequence:
    Capture attention → clarify the problem → prove credibility → explain the solution → reduce objections → convert.

# Execution prompt for Codex

When asked to create or improve a SaaS landing page, follow this process:

1. Identify the SaaS type:
   - self-serve;
   - demo-led B2B;
   - early-stage;
   - complex platform.

2. Extract or infer:
   - target user;
   - main problem;
   - desired result;
   - core mechanism;
   - proof available;
   - CTA model;
   - pricing or access model.

3. Build the page with this structure:
   - Sticky navbar
   - Hero
   - Initial social proof or credibility
   - Problem
   - Pain vs relief
   - Short proof block
   - Solution
   - How it works
   - Intermediate CTA
   - Product demo
   - Storytelling or credibility
   - Pricing or access
   - FAQ
   - Social proof wall if real proof exists
   - Final CTA
   - Footer

4. Keep copy outcome-driven.

5. Do not invent:
   - testimonials;
   - logos;
   - numbers;
   - pricing;
   - compliance;
   - guarantees.

6. Implement clean, responsive, maintainable code.

7. Run the available checks:
   - lint;
   - typecheck;
   - build;
   - responsive review;
   - link and CTA verification;
   - placeholder removal.

8. Deliver a concise summary of:
   - sections created;
   - assumptions made;
   - proof intentionally omitted;
   - commands run;
   - remaining risks or TODOs.
