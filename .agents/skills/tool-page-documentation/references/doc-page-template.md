# Tool/Page Documentation Template

Use this outline when documenting a tool, widget, or page.

## Suggested structure

### 1. Intro
- what this tool is
- why it exists
- who it helps

### 2. À quoi sert cet outil
- the real job-to-be-done
- what decision or action it supports

### 3. Ce que vous voyez dans la carte / page
- score
- badge / verdict
- alerts
- dimensions
- confidence

### 4. Données utilisées
- subjective inputs
- objective inputs
- historical signals
- behavioral signals

### 5. Logique de lecture
- what a high result means
- what a medium result means
- what a low result means

### 6. Limites
- data quality issues
- missing data behavior
- non-medical / non-diagnostic boundaries

### 7. Comment obtenir le meilleur de l’outil
- better check-ins
- better logs
- better measurement cadence

### 8. Cas concrets
- good state
- fragile state
- bad state
- sparse-data state

### 9. Erreurs d’interprétation
- confusing score with certainty
- overreacting to one signal
- ignoring context

### 10. Que faire si le widget dit X
- if green / adapted
- if orange / fragile
- if red / not adapted
- if confidence low

## Tone rules

- Explain like a strong onboarding doc, not a spec.
- Keep paragraphs short.
- Prefer examples over abstract theory.
- Use the user’s vocabulary when possible.

## Product rule

If two nearby tools can be confused:
- explicitly clarify their difference
- explain when to use one versus the other
